# CLAWrity — Project Conventions

## Project Overview

CLAWrity is an OpenClaw skill that provides ADHD & Autism support as a conversational companion. It is built for the OpenClaw agent framework and distributed as an npm package via `npx clawrity@latest`.

## Git Conventions

### Commit Messages
- Use [Conventional Commits](https://www.conventionalcommits.org/) format
- Format: `<type>(<scope>): <description>`
- Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `build`
- Scope examples: `skill`, `cli`, `scripts`, `templates`, `config`
- Examples:
  - `feat(skill): add body double mode with timer-based check-ins`
  - `fix(cli): handle missing openclaw directory gracefully`
  - `docs(readme): add installation and usage instructions`
  - `chore(build): configure npm package metadata`
- **NEVER** include tool attribution (e.g., "written with X", "generated by Y") in commit messages, branch names, or code comments

### Branch Naming
- Feature branches: `feat/<feature-name>` (e.g., `feat/body-double-mode`)
- Fix branches: `fix/<issue-description>`
- Documentation: `docs/<topic>`
- Use kebab-case for branch names

### Pull Requests
- Always create feature branches from `main`
- Write descriptive PR titles and descriptions
- Reference related issues when applicable

## Code Style

### JavaScript/TypeScript
- Use `const` over `let`, avoid `var`
- Use template literals for string interpolation
- Prefer async/await over raw promises
- Add JSDoc comments for public functions
- No unused imports or variables

### Shell Scripts
- Use `set -euo pipefail` at the top
- Use shellcheck-compliant patterns
- Add descriptive comments for non-obvious logic

### Markdown (SKILL.md, templates)
- Use ATX-style headers (`#`)
- One blank line between sections
- Code blocks must specify language

## Project-Specific Rules

- All variable names, file names, and function names must be original — no reuse from other OpenClaw skills
- Environment variable names should be prefixed or clearly scoped to avoid conflicts
- The companion persona must be neurodivergent-affirming and never condescending
- When in doubt about a design decision, **always ask** — do not assume
